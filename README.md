# 내 청취 기록으로 만든 음악 추천 엔진

내 유튜브 뮤직 청취 기록(재생 2만 건·18개월)으로 만든 개인 음악 추천기입니다. 시드곡 하나를 고르면 어울리는 플레이리스트를 만들어 YouTube Music에 바로 등록합니다.

> **만든 이유.** 기존 유튜브 뮤직은 이미 질려서 스킵하는 곡까지 계속 추천을 합니다. "지금 이 곡을 얼마나 좋아하는지"를 제대로 재는 사용자의 호감 신호를 적극적으로 반영하는 추천기를 직접 만들어 보고 싶었습니다.

## 알고리즘 계산 한 줄 정리

**추천 점수 = 유사도 × 진폭(A) × (1 + 현재강도 R)<sup>β</sup>**

| 신호 | 의미 | 방법 |
|---|---|---|
| 유사도 | 시드곡과 결이 맞나 | MERT 음악 임베딩 코사인 |
| 진폭 A | *원래* 얼마나 사랑하나 | 행동 기반 LightGBM (호감도 정확도 r ≈ 0.64) |
| 현재강도 R | *지금* 얼마나 듣고 있나 | 최근 완주 빈도(시간 감쇠) |
| β | 익숙함(0) ↔ 현재 몰입(2) 조절 손잡이 | — |

추천기를 만들며 시작부터 세운 가설이 있습니다. 
— *"한 사람이 노래를 소비하고 질려가는 흐름은, 곡과 상관없이 보편적인 모형을 갖는다"* 

- 식어가는 **'모양'은 보편적**입니다. 서로 다른 두 방법으로 검증했습니다(제1주성분 83%, 귀무 대비 11σ).
- 그러나 **"언제 질릴지"는 원리적으로 미리 못 맞힙니다.** "지금 식는 중인지"만 맞힐 수 있습니다.
- 즉 예측에 쓸 수 있는 과거 청취 정보는 **두 숫자(사랑 A·현재강도 R)로 압축**됩니다. 그래서 추천기도 이 둘만 씁니다.

> Flask 웹앱(`app.py`)으로 동작합니다. 단, 청취 기록·라벨·임베딩 캐시는 개인정보라 저장소에 포함하지 않았습니다(코드 쇼케이스).

## 1. 문제 — 세 가지 판단으로 쪼개기

"지금 이 곡을 추천해도 될까?"에는 사실 서로 다른 판단이 섞여 있습니다.

| 판단 | 질문 | 신호 |
|---|---|---|
| ① 어울리나 | 시드곡과 결이 맞나? | 유사도 |
| ② 좋아하나 | *원래* 이 곡을 사랑하나? | 진폭 A |
| ③ 지금인가 | *지금* 들을 때인가? | 현재강도 R |

처음엔 ③을 "생애주기 곡선 위 위치" 같은 정교한 신호로 풀려 했지만, 검증 끝에 *예측에 쓸 수 있는 부분*은 단순한 현재강도(R)로 좁혀졌습니다(3장). 최종 점수는 세 신호의 곱입니다.

---

## 2. 진폭 A — "원래 얼마나 사랑하나"

- **정답지**: 곡 118개를 직접 1~5점으로 평가(모델의 목표값).
- **모델**: LightGBM. 정답지가 적고(~118곡) 입력이 표 형태 수치라, 과적합을 누르며 피처 상호작용(예: 같은 행동도 아티스트에 따라 의미가 다름)을 잡기에 적합합니다.
- **피처** — *'일부러 골랐다'는 신호* 중심(스킵 거의 안 하는 큐레이터형이라 의도적 선택이 호감도를 잘 드러냄):
  - 완주 재생수, 완주율·스킵률(재생 간격 30초 기준), 검색 후 재생 비율, 세션 시작 비율
  - 곡 음색을 요약한 오디오 특성
  - 아티스트 평균 점수 — 라벨 없는 곡(전체의 41%)은 모델 예측으로 채워 계산(준지도 학습)
  - *recency(최근 빈도)는 일부러 제외* — 그건 '지금의 행동'(R)이라 '원래의 사랑'과 분리합니다. 덕분에 추천식에서 A와 R을 함께 써도 이중 계산이 아닙니다.
- **평가**: 곡 단위 5-겹 교차검증을 10회 반복해 평균낸 호감도 정확도 → **r ≈ 0.64**.

> **정답지 교훈.** 4월·6월 두 번 매긴 라벨로 같은 모델을 만들었더니 6월이 r 0.73으로 더 높았지만, 6월은 아티스트 단위로 뭉뚱그린 라벨(eta² 0.56 vs 4월 0.38)이라 *강한 아티스트 피처에 우연히 맞아떨어진* 것이었습니다. 곡별로 섬세한 4월(0.64)을 채택했습니다 — 검증 점수가 높다고 더 옳은 정답지는 아닙니다.

> **유의.** 행동 피처들은 서로 상관이 큽니다. 트리 모델이라 예측 정확도엔 무해하지만, 개별 피처 중요도 해석은 신중해야 합니다.

## 3. 생애주기 — 핵심 발견

**'질림'의 정의**: 한 곡을 들을수록 다시 찾는 빈도가 점점 식는 것. 곡별 **누적 소비 곡선**(그 곡의 생애에 걸쳐 재생이 어떻게 분포하는지)으로 측정합니다.

### (1) 식는 '모양'은 보편적이다 — 두 방법으로 검증

- **정규화 함수형 PCA**: 곡(재생 16회 이상)마다 누적 소비 곡선을 그려 주성분분석. **제1주성분이 곡 간 차이의 83%**를 설명합니다(질림 없이 무작위로 흩뿌린 귀무모형은 61% = 브라운 다리 이론값과 일치, 실제 83%는 **11σ 위**). 평균적으로 생애 첫 25%에 재생의 42%가 몰립니다. 다른 사용자 7명 전원에서 재현됐습니다.
- **절대 시간 점과정(NHPP)**: 위 정규화는 x축 분모로 *마지막 재생일(미래)*을 써서 누수 위험이 있습니다. 그래서 날짜를 정규화하지 않고 재생 속도를 직접 모형화해 재검증했습니다. "모양을 모든 곡이 공유한다"는 모형이 정보량 기준(BIC)에서 우세 → **누수 없이도 보편성이 확인**됐습니다.

곡마다 다른 건 **깊이(진폭)와 수명(시간 스케일)**뿐, 식는 *방식*은 공통입니다.

### (2) 그런데 예측은 '지금'만 된다

- **"언제 질릴지"(수명) — 못 맞힙니다.** 곡의 수명 스케일은 초반 데이터로 추정이 거의 불가능합니다. 생애의 90%를 봐도 추정이 50% 틀립니다(앞부분만 보면 빠르게 식는 곡과 느리게 식는 곡이 구분되지 않으니까요). 원리적 한계입니다.
- **"지금 식는 중인지"(다음 28일 완주수) — 맞힙니다.** 과거 기록만으로(누수 0), 처음 보는 곡으로만 평가한 곡 단위 교차검증에서 **R² ≈ 0.39**. 다만 그 예측력의 대부분은 recency 관성이고, 호감도가 더하는 순증가는 작습니다(+0.03).

### (3) 결론 — 과거 청취는 두 숫자로 압축된다

봉우리 위치·곡선 모양 등에서 recency를 넘어서는 신호를 여러 각도로 찾았지만, 전부 **사랑(A)·현재강도(R)** 두 숫자로 환원됐습니다. 역설적으로, 모양이 *보편적*일수록(= 모든 곡이 똑같을수록) 곡을 *구별*해야 하는 추천에는 쓸모가 없습니다. 그래서 추천기는 모양 대신 이 두 숫자만 씁니다.

## 4. 유사도 — "시드와 어울리나"

손수 설계한 음향 특성(MFCC 등 37차원)으로 시작했지만, *음색*은 잡아도 *장르·분위기*를 못 잡아 록 시드에 옛 발라드가 섞였습니다. 태그·메타데이터·가사를 합친 다중 신호로 보완해도 한계가 남았고, 음악 이해용으로 사전학습된 모델 **MERT의 임베딩(768차원, 라이브러리 2,690곡)**으로 교체하자 장르 일관성이 크게 올랐습니다(록 시드엔 록, 인디 시드엔 인디).

핵심 차이는 표현을 만든 주체입니다 — 1단계는 *사람이 무엇을 잴지 직접 설계*한 특성이고, MERT는 *모델이 대량의 음악을 학습하며 스스로 만든* 표현입니다. 후자가 장르·스타일 같은 고수준 특징을 더 잘 담습니다.

## 5. 추천 생성

시드곡과 세 가지 설정(① 플레이리스트 곡 수, ② 그중 새로 듣는 곡 수, ③ 모드)을 정하면:

```
1) 유사도 관문 — MERT 임베딩으로 시드와 가장 비슷한 곡을 후보로 추림
2) 본편(아는 곡) — 점수 = 유사도 × A × (1+R)^β 로 랭킹해 (곡 수 − 신곡 수)곡
3) 신곡 슬롯 — 시드와 비슷하지만 거의 안 들은 곡을 예측 호감도로 골라 (신곡 수)곡
4) 가중 랜덤 샘플링으로 합침 (매번 조금씩 다른 리스트, 시드곡이 1번)
```

손잡이는 β 하나입니다(α는 1 고정).

| 모드 | β | 성격 |
|---|---|---|
| 편안 | 0 | 순수 호감도 — 원래 가장 사랑하는 곡 |
| 균형 | 1 | 사랑과 현재 강도를 고르게 |
| 지금 기분 | 2 | 지금 빠져 있는 곡 위주 |

신곡 수는 *아는 곡 ↔ 모르는 곡*(explore), 모드는 *익숙함 ↔ 현재 몰입*(exploit) 축으로, 추천의 고전적인 두 손잡이입니다.

## 6. 한계와 다음 단계

- 정답지가 118곡으로 적어 진폭 정확도의 천장이 여기서 옵니다. MERT로도 가끔 결이 안 맞는 곡이 남습니다.
- 현재강도 R의 예측력은 대부분 recency 관성이라, *성능 지표*가 아니라 *편성 정책*으로만 씁니다.
- **진짜 검증은 못 했습니다.** 본질적으로 직접 들어 좋은지를 봐야 하는데, 저작권상 출시가 불가능하고 라벨링 부담 탓에 사용자 모집도 못 해 *2달간 직접 사용*한 게 전부입니다(써본 느낌은 좋았으나 객관적 근거는 아닙니다).
- **제대로 출시·검증한다면**: ① 수동 호감도 라벨을 *행동 신호 + 가벼운 온보딩(능동 학습)*으로 대체하고, ② A/B 비교·세션 지표·명시적 피드백으로 만족도를 측정해야 합니다. 거기가 다음 단계입니다.
